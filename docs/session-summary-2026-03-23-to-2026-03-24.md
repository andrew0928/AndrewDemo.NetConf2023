### 時序摘要

1. 一開始是高層架構評估。
   - 你提出的核心需求是：
     - 單一 code base，單一部署只服務單一 shop，但可依 `shop-id` 切換組態
     - `DiscountRule` 模組化
     - `ProductService` 模組化，支援非傳統實體商品，例如預約服務
   - 回應方向先收斂為：
     - `ShopManifest` / `ShopRuntime` / 啟動時組態
     - discount plugin 邊界
     - product service 邊界
     - 後續應把 orchestration 從 controller 拉回 core

2. 你很早就把開發流程明確 phase 化。
   - Phase 1:
     - 先出 `/spec`
     - 先出 `/spec/testcases`
     - 先出 `/src/AndrewDemo.NetConf2023.Abstract`
     - 此階段可以修改 spec 與 contract
   - Phase 2:
     - 依已定案的 spec / contract 重構 `.Core` 與 host
     - agent 高自主開發、自我驗證
     - 你只在 review 時拍板
   - `/docs` 跟 `/spec` 的用途也在這一段被明確切開。

3. 你先處理技術前置與範圍控制。
   - 若需要可直接升級到 `.NET 10` 與 `.slnx`
   - 不需要 runtime backward compatibility
   - `ConsoleUI` 暫時不納入主線，只保留可查閱

4. 第一波實作集中在 shop runtime 與 discount plugin Phase 1。
   - 先建立 branch 與前置 commit
   - 你要求先看 `.Abstract` class diagram 與命名說明
   - 你對命名做了連續幾次收斂：
     - `DiscountApplication` 改成 `DiscountRecord`
     - `IDiscountRulePlugin` 改成 `IDiscountRule`
     - `DiscountCartLine` 改成 `CartLineItem`
   - 你接著要求用 sequence diagram 驗證 discount engine 在真實購物情境中的互動方式

5. discount domain 在這個 session 中被大幅收斂。
   - 你明確要求 discount domain 只負責「試算立即結帳可得的折扣」
   - 直接相關的必要物件只保留最小集合
   - 其他搭配物件，例如 cart line、cart context，回到 cart domain
   - `CartContextSnapshot` 被視為可疑設計，要求重新評估是否真的必要
   - `IShopRuntimeContext`、factory 與 database context 是否過度散落，也被要求重審
   - 之後又繼續收斂：
     - `CartContextBuilder` 更名為 `CartContextFactory`
     - `IDiscountEngine` 若無必要則移除
     - `Cart` 與 `CartContext` 保留區隔，但 line item 盡量共用、偏 immutable / readonly

6. runtime 使用方式與 API 手動驗證工具也被納入要求。
   - 你問 API 啟動時怎麼指定 `shop-id`
   - 你要求更新 `oauth2.http`，讓 VS Code REST Client 能自動取得 token
   - 當 REST Client 對 redirect header 支援不足時，你拍板：
     - API 在 develop mode 額外回 `X-OAuth-Code`

7. 第二波主題切到 Product Phase 1。
   - 你先用業務語言描述最終目標：
     - 可以販售動態產生的 product
     - 結帳成功後 `ProductService` 會收到交易結果事件
     - fulfillment 不等於 order complete
     - 取消允許存在
   - 同時你強調邊界：
     - reservation 查詢、預約前確認、預約後取消等流程，不放進 shared `Abstract`
     - 商店自行整合 product service，主系統不碰那些外部流程
     - database extension 可以 side-by-side，不強迫 shared payload

8. Product domain 的確認點在 session 中被逐步拍板。
   - 每個 shop 只啟用一個 `IProductService`
   - `ProductId` 改成 `string`
   - published products 與 hidden products 要能共存
   - product callback 單位要是 order-scoped event，而不是 raw `Order`
   - order complete 與 fulfillment result 分離
   - callback 失敗不推翻 order complete
   - cancel 允許 partial cancel，event 需能指出 affected lines
   - extension data 走 side-by-side，不進 shared contract

   - phase 0:
     - 以 source / entry point / 測試 / sample 為主

     - 以 `spec`、`.Abstract`、source、tests 為主
